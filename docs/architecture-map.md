# Architecture Map — Knowledge Management Kit

Карта системы от ядра наружу. Каждый слой отвечает на вопрос: **"Какую проблему это решает и почему предыдущего слоя недостаточно?"**

---

## Layer 0 — Ядро: CLAUDE.md + decisions/

```
CLAUDE.md (brief + правила)
meta/decisions/{domain}/{CODE}.md (ADR-файлы)
```

**Проблема:** Claude Code теряет всё между сессиями. Каждый разговор — с нуля.

**Решение:** Два типа файлов. CLAUDE.md читается автоматически при старте — это точка входа. Секция "Модель проекта" (brief) даёт рабочую ментальную модель за одно прочтение. Decisions файлы хранят каждое решение в ADR-формате: что решили, почему, зависимости, условия пересмотра.

**Принцип:** Решили → файл. Не "потом запишу". Git — единственная система версий.

**Когда достаточно:** Проект маленький, решений < 10, один домен. Claude открывает CLAUDE.md, видит brief, по ссылке читает нужный файл.

**Когда ломается:** Решений > 10. Brief не может ссылаться на каждое. Claude не знает что искать и где.

## Layer 1 — Навигация: индексы + roadmap + sessions

```
meta/decisions/_index.md          (hub — таблица доменов, счётчики)
meta/decisions/{domain}/_index.md (доменный скелет — все записи)
meta/docs/_index.md               (hub исследований)
meta/docs/{topic}/_index.md       (доменный индекс исследований)
meta/roadmap.md                   (стек задач, статусы, глубины)
meta/sessions.md                  (сессионный контекст — отдельно от задач)
```

**Проблема:** При 10+ решениях Claude не знает, что существует и где лежит. Brief не масштабируется как навигатор.

**Решение:** Трёхуровневая адресная модель:
1. **Brief** (CLAUDE.md) — понимание проекта, ≤120 строк
2. **Hub + roadmap** — навигация: какие домены, сколько решений, что в работе
3. **Доменный индекс** — скелет: все записи домена с тегами и связями

Ленивая загрузка: при старте Claude читает только brief + hub. Доменные индексы — когда задача касается конкретного домена. Сами файлы решений — по ссылкам из индекса.

Roadmap отделён от sessions: задачи (что делаем) ≠ сессионный контекст (что было в прошлой сессии, что осталось). Sessions — слияние блоков, каждая сессия добавляет свой.

**Когда достаточно:** Claude может найти любое решение за 2 шага (hub → доменный индекс → файл). Roadmap показывает что в работе.

**Когда ломается:** Длинные сессии. Claude забывает начало разговора. Решения тонут в промежуточном шуме. При /compact теряется незаписанный контекст.

## Layer 2 — Внимание: FAR-протокол

```
Full Attention Residuals.md       (спецификация)
.claude/commands/far.md           (команда /far)
```

**Проблема:** В длинной сессии контекст засоряется: промежуточные шаги, тупиковые ветки, обработанные логи. Claude тратит внимание на мусор. При /compact — механическое сжатие, непредсказуемые потери.

**Решение:** FAR — проактивное управление вниманием. Три слоя:
- **HOT** (max 3-5) — текущая задача, открытые вопросы
- **WARM** — принятые решения, результаты, инсайты (тезисно)
- **COLD** — мусор, безопасно забыть

Ключевое отличие от /compact: FAR работает **до** переполнения, семантически (не механически), и смотрит **вперёд** (секция "Горизонт").

Триггеры: смена фазы, закрытие подзадачи, 8-12 обменов без аудита, изменение направления. Или вручную: `/far`.

Связка FAR → sessions.md: WARM из аудита записывается в сессионный контекст при коммите. Так знания из длинной сессии переживают и compact, и завершение сессии.

**Когда достаточно:** Claude управляет своим вниманием, не теряет нить, не переоткрывает решённое.

**Когда ломается:** Человек забывает запускать /far. Коммитит без обновления индексов. Не записывает решения. Система зависит от дисциплины, которую легко нарушить.

## Layer 3 — Автоматизация: хуки

```
.claude/hooks/
├── pre-commit-secretary.sh       # PreToolUse (git commit)
├── session-start-recovery.sh     # SessionStart
├── pre-compact-handoff.sh        # PreCompact
├── post-compact-reload.sh        # PostCompact
├── rebuild-index.sh              # ручной
└── lint-refs.sh                  # ручной
```

**Проблема:** Протоколы работают только если их соблюдать. Claude — языковая модель. Она не будет проактивно проверять индексы перед коммитом, если не напомнить.

**Решение:** 6 хуков, привязанных к событиям жизненного цикла:

| Хук | Событие | Что делает |
|-----|---------|------------|
| **pre-commit-secretary** | Перед git commit | Показывает секретарский чеклист (7 пунктов). Проверяет: обновлены ли индексы, roadmap, sessions. Ищет orphan-файлы. |
| **session-start-recovery** | Старт сессии | Проверяет незакоммиченные изменения, свежесть roadmap, загружает последний сессионный блок. |
| **pre-compact-handoff** | Перед /compact | Если WARM не записан → CRITICAL предупреждение. Если записан → мягкое напоминание проверить полноту. |
| **post-compact-reload** | После /compact | Инжектит стек задач из roadmap + последний сессионный блок обратно в контекст. |
| **rebuild-index** | Ручной запуск | Аварийное восстановление: парсит ADR-файлы, генерирует доменные _index.md и hub. Суть не восстанавливается — ставит заглушки. |
| **lint-refs** | Ручной запуск | Валидация: ссылочная целостность [[CODE]], контракт ADR, синхрон файл↔индекс, orphan-теги. Advisory mode — не блокирует. |

Хуки не выполняют работу за Claude — они **напоминают** и **проверяют**. pre-commit показывает чеклист, а не редактирует файлы.

**Когда достаточно:** Система самоподдерживающаяся. Ничего не потеряется молча. Даже если Claude забудет — хук напомнит.

**Когда ломается:** Хуки специфичны для Claude Code (settings.local.json). Другие LLM-клиенты не поддерживают.

## Layer 4 — Инструменты: /context + граф зависимостей

```
.claude/commands/context.md       (команда /context)
.claude/scripts/context.py        (движок)
meta/_tags.md                     (словарь хештегов)
agents/AGENT_PROTOCOL.md          (протокол ролей)
meta/project_manifest.md          (карта файловой структуры)
```

**Проблема:** Решения не изолированы — они связаны. Изменение одного может сломать другое. При 20+ решениях граф зависимостей не помещается в голову.

**Решение:** context.py строит карты зависимостей по трём осям:

- `/context CODE` — forward graph: прямые (1 hop) и транзитивные (2 hop) зависимости + тематические пересечения (общие теги ≥ 2)
- `/context CODE!` — reverse graph: кто зависит от этого решения (impact analysis)
- `/context #tag` — все решения с тегом (тематический срез)

Теги из словаря `_tags.md` — общего для decisions и docs. Новый тег → сначала в словарь.

AGENT_PROTOCOL.md определяет роли (Архитектор, Планировщик) и протокол фиксации решений. Манифест — карта файловой структуры, загружается по запросу.

## Обзор: вложенность слоёв

```
┌─────────────────────────────────────────────────┐
│  Layer 4: Инструменты                           │
│  /context, /far, agents, manifest               │
│  "Как навигировать граф зависимостей"            │
├─────────────────────────────────────────────────┤
│  Layer 3: Автоматизация                         │
│  6 хуков: pre-commit, session-start,            │
│  pre/post-compact, rebuild, lint                 │
│  "Как не забыть процесс"                        │
├─────────────────────────────────────────────────┤
│  Layer 2: Внимание                              │
│  FAR: HOT/WARM/COLD + Горизонт                  │
│  "Как не утонуть в длинной сессии"              │
├─────────────────────────────────────────────────┤
│  Layer 1: Навигация                             │
│  Hub/доменные индексы, roadmap, sessions         │
│  "Как найти нужное среди многого"               │
├─────────────────────────────────────────────────┤
│  Layer 0: Ядро                                  │
│  CLAUDE.md (brief) + decisions/ (ADR)            │
│  "Как помнить между сессиями"                   │
└─────────────────────────────────────────────────┘
```

Каждый следующий слой нужен только когда предыдущий перестаёт справляться. Маленький проект с 5 решениями может обойтись Layer 0–1. Полный стек — для проектов с десятками решений и длинными сессиями.

## Потоки данных

### Старт сессии
```
SessionStart hook
  → проверяет uncommitted changes
  → проверяет свежесть roadmap
  → загружает последний блок sessions.md
Claude читает CLAUDE.md (brief + правила)
  → по необходимости: hub → доменный index → файлы
```

### В процессе работы
```
Решение принято
  → ADR-файл → доменный _index → hub _index
  → если меняет brief → обновить CLAUDE.md

FAR-аудит (автотриггер или /far)
  → HOT в фокусе, WARM тезисно, COLD забыть
  → WARM → sessions.md (при коммите)
```

### Коммит
```
pre-commit-secretary hook
  → секретарский чеклист (7 пунктов)
  → проверка: обновлены ли индексы, roadmap, sessions
  → поиск orphan-файлов
  → Claude исправляет пропущенное
  → коммит
```

### Компрессия контекста
```
pre-compact-handoff hook
  → WARM записан? → мягкое напоминание
  → WARM НЕ записан? → CRITICAL: запиши сейчас

/compact выполняется

post-compact-reload hook
  → инжектит стек задач + последний сессионный блок
  → Claude продолжает с контекстом
```
